# Отчёт по проекту: Система вопросно-ответного взаимодействия на основе RAG для документации CAD-ядра

---

## 1. Введение и постановка задачи

Целью данного проекта является разработка интеллектуальной системы вопросно-ответного взаимодействия (QA-система) для технической документации по геометрическим ядрам OpenCascade Technology (OCCT) и Analysis Situs. Система предназначена для помощи разработчикам в поиске ответов на технические вопросы, связанные с B-Rep топологией, алгоритмами булевых операций, форматами обмена данными (STEP, IGES) и другими аспектами CAD-разработки.

Ключевая проблема, которую решает данная система, — это «галлюцинации» (hallucinations) больших языковых моделей (LLM). При прямом обращении к LLM без контекста модель склонна генерировать правдоподобно звучащие, но фактически неверные ответы, особенно в узкоспециализированных технических областях. Архитектура Retrieval-Augmented Generation (RAG) решает эту проблему, принудительно ограничивая ответ модели только теми фактами, которые были извлечены из верифицированной базы знаний.

---

## 2. Архитектура системы

Система реализована в виде конвейера из нескольких последовательных этапов.

### 2.1 Построение базы знаний (Offline-этап)

Исходные документы в формате Markdown (документация OCCT и Analysis Situs) загружаются из локальной директории и разбиваются на фрагменты (chunks) с помощью `RecursiveCharacterTextSplitter` со следующими параметрами:
- Размер фрагмента: 800 символов
- Перекрытие: 150 символов

Для каждого фрагмента вычисляется векторное представление (embedding) с использованием модели `BAAI/bge-small-en-v1.5`. Полученные векторы сохраняются в локальной векторной базе данных ChromaDB. Итоговая база содержит **3900 фрагментов**.

### 2.2 Гибридный конвейер извлечения (Online-этап)

При поступлении запроса от пользователя система выполняет следующую последовательность действий:

**Шаг 1 — Предварительный перевод запроса (Query Translation).**
Поскольку документация написана на английском языке, а пользователь может задавать вопросы на китайском, система использует LLM для предварительного перевода запроса на английский язык. Это критически важный шаг для обеспечения работоспособности как векторного поиска, так и BM25.

**Шаг 2 — Двухканальный поиск (Dual Retrieval, Top-50 каждый).**
- *Плотный поиск (Dense Retrieval)*: семантический поиск по косинусному сходству векторов в ChromaDB.
- *Разреженный поиск (Sparse Retrieval)*: лексический поиск по алгоритму BM25, учитывающий частоту терминов.

**Шаг 3 — Слияние результатов (RRF Fusion).**
Результаты двух каналов объединяются методом Reciprocal Rank Fusion (RRF) с параметром сглаживания k=60. Формула: `score += 1 / (k + rank + 1)`. Метод устойчив к различным масштабам оценок двух каналов.

**Шаг 4 — Повторное ранжирование (Cross-Encoder Reranking).**
Топ-50 объединённых результатов передаются в модель `cross-encoder/ms-marco-MiniLM-L-6-v2`, которая оценивает релевантность каждой пары (запрос, фрагмент) совместно, что обеспечивает более точное ранжирование по сравнению с bi-encoder подходом.

**Шаг 5 — Генерация ответа.**
Топ-5 наиболее релевантных фрагментов передаются в LLM (DeepSeek Chat) в качестве контекста. Промпт явно запрещает модели использовать знания, выходящие за рамки предоставленного контекста, что минимизирует риск галлюцинаций.

## 3. Оценка качества системы

### 3.1 Цель оценки

Поскольку качество RAG-системы определяется прежде всего точностью извлечения контекста, а не литературными достоинствами генерируемого текста, оценка сосредоточена исключительно на **качестве конвейера извлечения (Retrieval Pipeline)**. В качестве основной метрики используется **Hit Rate @ 5** — доля запросов, для которых правильный фрагмент оказался среди пяти наиболее релевантных результатов.

### 3.2 Формирование тестового набора данных

Для объективной оценки был сформирован «золотой» тестовый набор (Golden Dataset) методом автоматической генерации с намеренным устранением эффекта утечки данных (Data Leakage).

Процедура формирования:

1. Из ChromaDB случайным образом отобраны **20 фрагментов** с длиной не менее 200 символов (из 7749 валидных фрагментов).
2. Для каждого фрагмента LLM сгенерировал один технический вопрос на английском языке в стиле реального разработчика — с явным запретом копировать фразы из исходного текста и упоминать «документ» или «отрывок». Это принципиально важно: наивный подход (генерация вопроса прямым перефразированием) создаёт высокое лексическое и семантическое пересечение между вопросом и ground truth, что искусственно завышает результаты векторного поиска и делает оценку бессмысленной.
3. Сформированные пары `(вопрос, эталонный фрагмент)` сохранены в файл `eval_dataset.json`.

Примеры сгенерированных вопросов:
- *"How do I compute the closest point on a 3D surface from a given 3D point using OCCT?"*
- *"How do I safely modify visual properties of an AIS interactive object without triggering a full redraw?"*
- *"How do I disable inverted volume checking in a boolean operation when I'm sure the inputs are valid?"*

Данный подход имитирует реальные поисковые запросы разработчиков и обеспечивает корректную оценку способности системы к семантическому обобщению.

### 3.3 Результаты оценки

Для демонстрации вклада каждого компонента архитектуры проведено сравнительное тестирование трёх конфигураций конвейера на одном и том же тестовом наборе:

| Конфигурация конвейера | Попаданий | Hit Rate @ 5 |
|------------------------|-----------|--------------|
| Только векторный поиск (Dense-only) | 14 / 20 | **70.0%** |
| Только BM25 (Sparse-only) | 10 / 20 | **50.0%** |
| Гибридный поиск + Cross-Encoder (Hybrid + Rerank) | 17 / 20 | **85.0%** |

**Анализ результатов.**

Гибридный конвейер превосходит чисто лексический поиск BM25 на **+35 процентных пунктов** и чисто векторный поиск на **+15 процентных пунктов**. Это убедительно подтверждает, что двухканальная архитектура не является избыточной: каждый из каналов компенсирует слабые места другого.

Детальный анализ расхождений между каналами наглядно демонстрирует взаимодополняемость подходов:

- Вопросы №1, №3, №6, №18 — BM25 нашёл правильный фрагмент, векторный поиск — нет. Это типичные случаи, когда вопрос содержит специфические технические термины (например, названия классов OCCT или параметров конфигурации), точное совпадение которых критично, а семантическое обобщение вектора уводит поиск в сторону.
- Вопросы №2, №5, №10, №11 — векторный поиск нашёл, BM25 — нет. Здесь вопрос сформулирован в общих словах разработчика, без точных ключевых слов из документации, что является сильной стороной семантического поиска.
- Гибридный конвейер с Cross-Encoder успешно объединил оба сигнала, дополнительно исправив случаи, где оба одиночных канала давали неверный результат (вопросы №15, №17, №19, №20).

### 3.4 Анализ граничных случаев и ограничений системы

В ходе разработки был выявлен критический граничный случай, связанный с **межъязыковым информационным поиском (Cross-Lingual Information Retrieval, CLIR)**.

**Проблема.** При подаче запроса на китайском языке без предварительного перевода (например, «如何实现放样» — «Как реализовать лофтинг») оба канала поиска полностью теряют работоспособность:
- BM25 присваивает нулевой балл всем документам, поскольку китайские иероглифы не пересекаются ни с одним токеном английской документации.
- Модель `bge-small-en-v1.5` является монолингвальной (обучена только на английском языке) и не обладает способностью к межъязыковому семантическому выравниванию. Вектор китайского запроса не коррелирует с векторами английских фрагментов, содержащих слово «Lofting».

**Следствие.** Система возвращает нерелевантные фрагменты, LLM не находит ответа в контексте и корректно отвечает «информация не найдена в базе знаний» — механизм защиты от галлюцинаций срабатывает правильно, однако полезность системы для пользователя равна нулю.

**Реализованное решение.** В производственной версии системы (`app.py`) введён обязательный этап предварительного перевода запроса (Query Translation) с использованием LLM перед передачей в конвейер извлечения. Это позволяет корректно обрабатывать запросы на китайском языке.

**Перспективные улучшения.** Для устранения зависимости от LLM-перевода рекомендуется замена базовой модели эмбеддингов на многоязычную, например `BAAI/bge-m3`, которая поддерживает более 100 языков и обеспечивает нативное межъязыковое семантическое выравнивание без дополнительных шагов.

## 4. Технический стек

| Компонент | Технология |
|-----------|------------|
| Векторная база данных | ChromaDB |
| Модель эмбеддингов | BAAI/bge-small-en-v1.5 |
| Разреженный поиск | BM25 (rank-bm25) |
| Модель повторного ранжирования | cross-encoder/ms-marco-MiniLM-L-6-v2 |
| Большая языковая модель | DeepSeek Chat (совместимый с OpenAI API) |
| Фреймворк оркестрации | LangChain |
| Пользовательский интерфейс | Streamlit |

---

## 5. Заключение

В рамках проекта разработана и верифицирована RAG-система для технической документации CAD-ядра. Реализованный гибридный конвейер извлечения достигает **Hit Rate @ 5 = 85%** на корректно сформированном тестовом наборе из 20 вопросов в стиле реальных запросов разработчиков, что на 35 процентных пунктов превышает результат чисто лексического поиска BM25 и на 15 процентных пунктов — чисто векторного поиска. Выявленное ограничение в области межъязыкового поиска устранено на уровне архитектуры посредством предварительного перевода запросов, а долгосрочным решением является переход на многоязычную модель эмбеддингов.
