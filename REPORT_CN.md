> 技术主题：检索增强生成（RAG）

---

## 一、什么是 RAG？

### 1.1 背景：大模型为什么会“胡说”？

大语言模型（LLM，如 DeepSeek、GPT 等）在通用对话上能力很强，但在**专业领域**（例如 CAD 几何内核、OpenCascade API）直接提问时，容易出现 **幻觉（Hallucination）**：回答听起来很专业，但类名、参数或步骤可能是编造的。

根本原因是：模型的“知识”来自训练语料，**并不保证包含你手头的官方文档**，也无法保证与当前版本一致。

### 1.2 RAG 的核心思想

**RAG（Retrieval-Augmented Generation，检索增强生成）** 的做法是：

1. **先把可信资料建成知识库**（本项目中是 OCCT、Analysis Situs 的 Markdown 文档）；
2. **用户提问时，先从知识库里检索最相关的片段**；
3. **只把这些片段作为上下文交给 LLM 生成答案**。

可以概括为一句话：

> **不让模型“凭记忆答题”，而是让它“先查资料再答题”。**

### 1.3 RAG 的典型流程

```
离线阶段（建库）                    在线阶段（问答）
─────────────────                  ─────────────────
原始文档 → 切分(Chunk)              用户问题
    ↓                                   ↓
Embedding 向量化                    检索 Top-K 相关片段
    ↓                                   ↓
存入向量数据库                      拼进 Prompt → LLM 生成答案
```

本项目的在线阶段比“单向量检索”更复杂，采用了 **混合检索 + 重排 + 查询翻译**，这在后文会详细说明。

### 1.4 理解 RAG 时需要区分的两个环节

| 环节 | 作用 | 本项目中的实现 |
|------|------|----------------|
| **检索（Retrieval）** | 找对文档片段 | 向量检索 + BM25 + RRF 融合 + Cross-Encoder 重排 |
| **生成（Generation）** | 用片段组织成自然语言回答 | DeepSeek Chat + 严格 Prompt（禁止编造） |

**检索质量往往决定 RAG 的上限**：检索错了，再强的模型也只能基于错误上下文回答。因此本项目专门用 `step3_evaluation.py` 评估检索命中率，而不是只凭“回答读起来顺不顺”。

## 二、本项目是如何一步一步完善的

项目采用 **分步脚本 + 最终 Web 应用** 的方式迭代，每一步都可独立运行和验证，便于学习和调试。

### 阶段 0：问题定义与数据准备

- **目标**：为 OpenCascade（OCCT）和 Analysis Situs 开发者提供基于**官方文档**的问答助手。
- **数据**：将 Markdown 文档放入 `data/raw_docs/`（英文技术文档）。
- **动机**：CAD 内核 API 繁多、概念抽象（B-Rep、布尔运算、STEP/IGES 等），传统关键词搜索难以理解“意思相近但用词不同”的问题。

---

### 阶段 1：构建知识库 — `step1_build_db.py`

**做什么：**

1. 扫描 `data/raw_docs` 下所有 `.md` 文件；
2. 用 `RecursiveCharacterTextSplitter` 切分：
   - `chunk_size = 800` 字符
   - `chunk_overlap = 150` 字符（相邻块有重叠，避免答案被截断在块边界）
3. 用 Embedding 模型 `BAAI/bge-small-en-v1.5` 将每个 chunk 转为向量；
4. 写入本地 **ChromaDB**（`./chroma_db`）。

**这一步我学到了什么：**

- RAG 不是“把整本书塞进模型”，而是 **Chunk + 向量索引**；
- 切分粒度影响检索效果：太大则噪声多，太小则上下文不完整。

**运行方式：**

```bash
python step1_build_db.py
```

---

### 阶段 2：混合检索管道 — `step2_hybrid_retrieval.py`

在接入界面前，先验证 **“能不能搜到对的片段”**。

**检索流水线：**

```
英文查询
   ├─ Dense（向量语义检索）Top-50
   └─ Sparse（BM25 关键词检索）Top-50
           ↓
      RRF 融合（Reciprocal Rank Fusion, k=60）
           ↓
   Cross-Encoder 重排（ms-marco-MiniLM-L-6-v2）
           ↓
        输出 Top-5 片段
```

**为什么不用一种检索就够？**

| 方式 | 擅长 | 弱点 |
|------|------|------|
| **向量检索（Dense）** | 语义相近、换种说法也能找到 | 对精确类名、参数名可能不敏感 |
| **BM25（Sparse）** | 精确匹配技术术语、类名 | 问法与文档用词差太多时容易漏检 |
| **混合 + 重排** | 综合两路优点，再用 Cross-Encoder 精排 | 计算稍多，但准确率更高 |

**RRF 的作用：** 把两路不同尺度的分数融合成统一排名，公式为  
`score += 1 / (k + rank + 1)`，对排名鲁棒、实现简单。

**运行方式：**

```bash
python step2_hybrid_retrieval.py
```

脚本内硬编码了一个英文测试问题，打印 Top-5 片段及来源文件，方便肉眼检查。

---

### 阶段 3：量化评估 — `step3_evaluation.py`

**目的：** 用数据证明混合检索确实更好，而不是“感觉上更好”。

**测试集生成（避免数据泄漏）：**

1. 从 ChromaDB 随机抽 20 个长度 ≥ 200 的 chunk；
2. 让 LLM **模拟真实开发者** 为每个 chunk 写一个问题：
   - 禁止照抄原文；
   - 禁止出现 “document / excerpt” 等字样；
3. 保存为 `eval_dataset.json`，形成 `(question, ground_truth)` 对。

> **重要改进（见 git 提交）：** 早期若用“容易被向量直接搜到的 chunk”生成问题，会**虚高**评估分数。后来修正为随机抽样 + 严格 prompt，使评估更接近真实场景。

**评估指标：Hit Rate @ 5**

- 对每个问题检索 Top-5；
- 若 Top-5 中任一片段与 `ground_truth` 高度相似（包含或 SequenceMatcher ≥ 0.5），算 **命中**；
- Hit Rate = 命中数 / 总题数。

**三路对比结果：**

| 检索方式 | 命中 | Hit Rate @ 5 |
|----------|------|----------------|
| 纯向量（Dense-only） | 14/20 | 70.0% |
| 纯 BM25（Sparse-only） | 10/20 | 50.0% |
| **混合 + Cross-Encoder 重排** | **17/20** | **85.0%** |

结论：混合检索比纯向量 **+15%**，比纯 BM25 **+35%**，说明双路架构在本场景下是有价值的。

**运行方式：**

```bash
python step3_evaluation.py
```

### 阶段 4：产品化界面 — `app.py`

在检索管道验证通过后，用 **Streamlit** 做聊天式问答界面。

**在线完整流程：**

```
用户中文提问
    ↓
LLM 翻译成英文（Query Translation，带缓存）
    ↓
混合检索（与 step2 相同逻辑）→ Top-5 上下文
    ↓
LLM 基于上下文流式生成中文回答
```

**Prompt 设计要点：**

- 明确要求 **仅根据上下文** 回答；
- 若上下文不足，必须回答“未在文档库中找到相关信息”，**严禁幻觉**。

**工程优化：**

- `@st.cache_resource` 缓存模型与数据库加载；
- `bm25_cache.pkl` 缓存 BM25 索引，避免每次冷启动重建；
- `translation_cache` 避免重复翻译相同问题。

**运行方式：**

```bash
streamlit run app.py
```

---

### 阶段 5：发现并解决的关键问题 — 跨语言检索（CLIR）

**现象：** 用户用中文问（如“如何实现放样”），若不翻译，检索几乎失效：

- BM25：中文与英文文档 **零词重叠**，得分全为 0；
- `bge-small-en-v1.5`：英文专用 Embedding，**无法对齐**中英文语义。

**解决：** 在 `app.py` 中增加 **查询翻译** 步骤，将中文问题转为英文后再检索。

**长期改进方向：** 换用多语言 Embedding（如 `BAAI/bge-m3`），可减少对翻译步骤的依赖。

## 三、给老师的讲解报告（书面版）

### 3.1 项目题目与意义

**题目：** 基于 RAG 的 CAD 几何内核文档智能问答系统  

**意义：**

- 解决 LLM 在专业 CAD 开发场景下的 **知识不可靠、易幻觉** 问题；
- 将 OpenCascade / Analysis Situs 文档转化为可检索、可问答的 **本地知识库**；
- 探索 **混合检索、重排序、跨语言查询** 在工程文档 RAG 中的实际效果。

### 3.2 我掌握的核心知识点

1. **RAG 的定义与动机**：检索 + 生成，用外部知识约束 LLM。
2. **文档预处理**：加载、切分、向量化、持久化到 ChromaDB。
3. **检索技术栈**：
   - Dense：语义向量检索；
   - Sparse：BM25 词频检索；
   - RRF：多路结果融合；
   - Rerank：Cross-Encoder 精排。
4. **评估方法**：构造 Golden Dataset、Hit Rate @ K、对比实验。
5. **工程与边界情况**：中英文跨语言、防幻觉 Prompt、缓存与性能。

### 3.3 系统架构（可向老师展示 README 中的图）

```
用户中文提问 → 翻译为英文 → 双路召回(各50) → RRF → 重排 → Top-5
                                                      ↓
                                            LLM 基于上下文生成回答
```

### 3.4 实验结论（用数字说话）

- 知识库规模：约 **3900+** 文本块（以实际 `step1` 运行输出为准）；
- 测试集：**20** 条模拟开发者风格的问题；
- **Hit Rate @ 5 = 85%**（混合检索），显著优于单路检索；
- 中文问答通过 **Query Translation** 保证检索可用。

### 3.5 局限与后续工作

| 局限 | 说明 | 改进方向 |
|------|------|----------|
| 文档语言 | 库内为英文 | 多语言 Embedding 或双语文档 |
| 评估规模 | 20 条样本 | 扩大测试集、引入 MRR、NDCG |
| 依赖在线 LLM | 翻译与生成需 API | 可换本地小模型做 ablation |
| 仅 Markdown | 未覆盖 PDF/HTML | 扩展文档加载器 |

### 3.6 总结陈述（可直接放在报告结尾）

本项目完整实现了一条 **“建库 → 检索 → 评估 → 应用”** 的 RAG 链路。我不仅搭建了可运行的问答系统，还通过对比实验证明 **混合检索优于单路检索**，并针对 **中英文跨语言** 问题给出了架构级解决方案。通过本项目，我理解了 RAG 不仅是“调用 API”，而是 **数据工程 + 信息检索 + 提示工程** 的组合。

**【开场，约 30 秒】**

老师好。我这次做的项目是一个 **基于 RAG 的 CAD 内核文档问答系统**，主要针对 OpenCascade 和 Analysis Situs 的开发文档。简单说，就是让开发者在查 API、查概念时，能像问同事一样用自然语言提问，但答案必须来自我们本地的官方文档，而不是模型瞎编。

---

**【第一部分：什么是 RAG，约 2 分钟】**

我先说明为什么需要 RAG。

大模型很强，但如果你直接问它“OCCT 里某个布尔运算怎么关 inverted volume 检查”，它可能会给出一个看起来合理、但实际上类名或参数不对的答案，这叫 **幻觉**。

RAG 的思路是：**先检索，再生成**。  
第一步，把文档切成很多小片段，做成向量数据库；  
第二步，用户提问时，先找出最相关的几个片段；  
第三步，只把这些片段塞进 Prompt，让模型根据资料回答。

所以 RAG 的本质是：**用检索到的证据约束模型的输出**。  
在这个项目里，我还特别注意了：检索做对了，后面生成才有意义，所以专门写了评估脚本，不只看回答漂不漂亮，而是看 **Top-5 里有没有找对文档片段**。

---

**【第二部分：项目怎么一步步做的，约 4 分钟】**

整个项目分四步，对应四个 Python 文件。

**第一步，`step1_build_db.py`，离线建库。**  
我把 `data/raw_docs` 里的 Markdown 读进来，按 800 字一块切开，用 `bge-small-en-v1.5` 做 Embedding，存进 ChromaDB。这一步解决的是“知识从哪来、怎么存”。

**第二步，`step2_hybrid_retrieval.py`，只做检索、不做界面。**  
我没有一开始就做网页，而是先把检索管道跑通。  
这里我用了一路 **向量检索** 抓语义，一路 **BM25** 抓关键词，然后用 **RRF** 把两路排名融合，再用 **Cross-Encoder** 对前 50 条重排，最后取 Top-5。  
这样做是因为向量检索和 BM25 各有短板：向量擅长“意思像”，BM25 擅长“类名、参数名对得上”，合在一起更稳。

**第三步，`step3_evaluation.py`，用数据验证。**  
我自动生成 20 个测试问题：从数据库里随机抽文档片段，让 LLM 扮演开发者提问，并且要求不能照抄原文，避免评估造假。  
指标是 **Hit Rate @ 5**：正确答案有没有出现在前 5 条检索结果里。  
结果是：纯向量 70%，纯 BM25 50%，混合加重排 **85%**。这说明我的混合方案是有实际提升的，不是堆概念。

**第四步，`app.py`，Streamlit 问答界面。**  
用户可以用中文提问。因为文档是英文的，我加了一步 **查询翻译**，把中文转成英文再检索，否则 BM25 和英文 Embedding 都对中文无效。  
生成时 Prompt 里写了：上下文没有就说“未找到”，**不允许编造**。界面上还会展示检索到的片段来源和重排分数，方便核对。

---

**【第三部分：我遇到的典型问题，约 1.5 分钟】**

开发过程中印象最深的是 **跨语言检索**。

我试过不翻译，直接中文搜“如何实现放样”，两路检索基本都废了：BM25 没有英文词匹配，英文 Embedding 也对不齐语义。  
翻译之后，同样的意思用英文 “lofting” 去搜，就能命中相关段落。  
这让我理解到：RAG 不只是算法，还要考虑 **语言和领域** 这些工程细节。

另外，测试集设计也改过：如果问题是从“很容易被搜到的 chunk”生成的，评估分数会虚高。后来我改成随机抽样加严格 prompt，评估才更可信。
